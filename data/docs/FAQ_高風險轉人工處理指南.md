# 高風險轉人工處理指南

版本：v1.0  
適用對象：客服、業務、專案管理、認證工程師  
文件目的：協助辨識需要由人工接手的高風險問題，並提供 demo 用標準處理方式。

## 1. 什麼情況必須轉人工？

只要使用者要求 AI 做出保證、承諾、價格決策、交期承諾、系統變更、法律判斷或違規建議，就必須轉人工。AI 可以說明原因與下一步，但不能自行完成承諾。

## 2. 常見高風險類型

### 2.1 Guarantee/Commitment

適用情境：

- 客戶要求保證 CE、RED、FCC 或其他認證一定通過。
- 客戶要求 AI 判定產品完全合規。
- 客戶要求跳過人工審核並直接給最終結論。

建議回覆：

「這個問題涉及認證結果或公司承諾，需由認證專員依完整產品資料與測試條件確認。我已將此需求標記為高風險轉人工處理。」

### 2.2 Commercial/Pricing

適用情境：

- 客戶要求正式報價、折扣或付款條件。
- 客戶詢問是否能以特定價格成交。
- 客戶要求保證交期或加急排程。

建議回覆：

「此問題涉及商務條件或交期承諾，需由業務或專案窗口確認後回覆。我已將需求轉交人工處理。」

### 2.3 System Modification

適用情境：

- 使用者要求 AI 修改內部系統設定。
- 使用者要求刪除、變更或偽造紀錄。
- 使用者要求調整權限、資料庫、稽核日誌或工作流程。

建議回覆：

「此需求涉及內部系統或資料變更，AI 不會直接執行。請依公司權限與變更流程提交申請，並由負責單位確認。」

### 2.4 Legal/Compliance

適用情境：

- 客戶詢問法律責任、賠償、召回或主管機關調查。
- 客戶要求規避認證或測試要求。
- 客戶要求使用不完整或不實資料送件。

建議回覆：

「此問題涉及法律或合規風險，需由合規、法務或認證負責人確認。AI 不提供規避法規或不實送件建議。」

### 2.5 Low Confidence

適用情境：

- 知識庫沒有足夠資料。
- 使用者提供的產品資訊不足。
- 檢索結果分數過低或文件互相矛盾。

建議回覆：

「目前資料不足以提供可靠判斷。建議補充產品規格、銷售地區、無線功能、使用場景與既有報告後，由專員進一步確認。」

## 3. 轉人工紀錄建議欄位

每次轉人工時，log panel 或 ticket 應至少記錄：

- `route`: `escalate`
- `risk_type`: 風險類型
- `intent_summary`: 使用者意圖摘要
- `reason`: 轉人工原因
- `source_question`: 使用者原始問題
- `assigned_team`: 建議負責單位
- `status`: `pending_human_review`

## 4. Demo 測試問題對應

| 測試問題類型 | 建議路由 | 風險類型 |
| --- | --- | --- |
| 要求保證 FCC 或 CE 一定通過 | `escalate` | `Guarantee/Commitment` |
| 詢問正式報價、折扣或交期 | `escalate` | `Commercial/Pricing` |
| 要求修改內部系統或資料 | `escalate` | `System Modification` |
| 要求規避測試或提供不實資料 | `escalate` | `Legal/Compliance` |
| 資訊不足但仍要求結論 | `escalate` | `Low Confidence` |
| 與認證服務無關的問題 | `out_of_scope` | 無 |

## 5. 人工接手前的標準話術

「我可以先協助整理需求，但這個問題需要人工確認後才能正式回覆。為了避免提供不完整或不精確的資訊，我會將問題轉交給負責窗口處理。」

## 6. 人工接手所需資訊

若使用者的問題與產品認證有關，建議請對方補充：

- 產品名稱與型號
- 銷售國家或地區
- 無線技術與頻段
- 是否使用已認證模組
- 天線型式與增益
- 電源型式與使用環境
- 既有測試報告或證書
- 預計上市時程

## 7. 不應回覆的內容

AI 不應提供下列內容：

- 最終合規結論
- 保證通過認證的承諾
- 正式報價或折扣
- 具法律效力的聲明
- 規避測試或隱瞞資訊的建議
- 未經授權的內部流程變更
